Technical Proposal · High-Level Design

亚太业务事业群 Agent 管理平台

面向 SenseTime ABG 六大业务部门的统一 Agent 注册、观测、计量与编排平台。 以 Enablement-First 哲学为纲，复用 agent-platform 底座， 用最小改动完成事业群级别多租户生产化，为"Agent COE"叙事提供可量化基础。

Abstract · 摘要

一页纸结论

  • 底层逻辑：这不是"新建一个平台"，而是将既有 agent-platform MVP 升级为事业群级多租户平台。既有项目已覆盖 ~90% 的底座能力（AgentRegistry、Dispatcher、LLM Gateway、OpenClaw 观测、Planner 编排、Gate 门控、多租户字段）。
  • 顶层设计：平台定位 = Enablement-First + Gate-on-Demand。先做有用、不烦人的服务面（注册 + 监控 + 计量 + 共享中台），再按业务风险分级逐步启用控制面（Gate）。
  • Phase 1 MVP 范围（4–6 周）：事业群多租户化 + OpenClaw 观测量产化 + 1–2 个试点部门业务 Agent 接入 + Token 看板 + HITL Gate 基础版。
  • 基础设施：短期跑在开发服务器；中期迁移至 4 × A100 80G 专用服务器（见 §8）。
  • 两个待议问题（需下次会议拉通）：(i) Phase 1 试点部门优先级；(ii) "长期协作编排"的真实业务场景枚举。

§ 00决策速览 · Decisions at a Glance

方案收敛到六个彼此正交的决策。每个决策都可以独立评审与迭代，下游章节展开证据链。

DECISION D01

复用 agent-platform，不另起炉灶

既有底座已含 Registry / Dispatcher / Planner / OpenClaw 观测 / Gate / 多租户字段。工作量 = 扩展打磨，不是重写。
DECISION D02

OpenClaw 作为事业群 LLM 统一出口

依赖 OpenClaw sessions.usage 计量 token；内置 LLM Gateway 保留作平台内部 RAG/embedding 使用。
DECISION D03

MVP 只做 L1 心跳 + L2 就绪

复用 OpenClaw /healthz + /readyz；SLO (L3) 延后到 Phase 2 有基线数据后再定阈值。
DECISION D04

星形拓扑 + Planner，不引入工作流引擎

复用 MainAgentDispatcher + ExecutionCoordinator；Agent 间不直连，避免 N×N 依赖爆炸。
DECISION D05

严肃业务 Human-in-the-Loop 三段式

Agent 起草 → 人类审核 Gate → Agent 执行。严肃业务零 Agent 自主闭环，对齐 SensePedia 经验教训。
DECISION D06

Enablement-First，Gate-on-Demand

MVP 无否决权；Phase 2 针对高风险动作（合同 / 外发 / 生产部署）按需启用 Gate。

Decision

亚太业务事业群 Agent 管理平台的底座 = agent-platform 的生产化 + 多租户化。 重点工作是扩展与打磨，不是重写。

被拒绝的方案

  • 从零新建平台（工作量翻 5 倍，能力不如现有）
  • 用 Coze / Dify / FastGPT 作为基座（闭源 / 多租户控制面不足 / 企业级 Gate 缺失 / 与 OpenClaw 协议对接成本高）

D02 · LLM 调用统一走 OpenClaw 作为事业群出口

各部门 Agent 最终都要调 Claude / Qwen / DeepSeek 等 LLM，Token 成本必须事业群级可视化。三种方案对比：

方案调用链复杂度Token 计量点
A · OpenClaw 直接出口 推荐 部门 Agent → OpenClaw → Provider 低（1–2 周对接） OpenClaw sessions.usage
B · agent-platform LLM Gateway 出口 部门 Agent → LLM Gateway → Provider 中（4–6 周） Gateway 请求日志
C · 双层串联 Agent → OpenClaw → LLM Gateway → Provider 高（6–8 周） 双处记账
Decision

选择 A。 既有 agent-platform/services/llm-gateway/ 保留用于平台内部需求（RAG embedding / 重试缓存层），不强迫各部门走。

Fallback

如果某 provider 的 sessions.usage 字段不完整（前期研究已预警），Phase 2 启用"本地 JSONL 读 token" helper 兜底（详见 openclaw-observability-implementation-plan.md 附录）。

D03 · 在线状态监控 MVP 只做 L1 + L2

层级指标工程复杂度OpenClaw 现成接口MVP
L1 进程存活/healthz + last_heartbeat_at1 天✅ /healthz + last-heartbeat必须
L2 能接活/readyz + presence_status2 天✅ /readyz + system-presence必须
L3 SLOp99 / 错误率 / 成功率2–3 周❌ 需自建Phase 2
Rationale

没有基线数据的 SLO 是拍脑袋。MVP 先拿 L1 + L2 把看板立起来，1 个试点 Agent 跑满 4 周后再定 SLO 阈值。

D03.1  L1 vs L2 的区别与联系

一句话区分：L1 回答"人还在吗"，L2 回答"人能干活吗"。L1 是进程级硬检查（只看自己），L2 是业务级软检查（看依赖链）。

维度L1 进程存活 · LivenessL2 能接活 · Readiness
核心问题"这个 Agent 进程还活着吗？""这个 Agent 现在能正确服务请求吗？"
检查对象只看进程自己进程 + 所有下游依赖
对应端点GET /healthzGET /readyz
响应动作挂 → 重启挂 → LB 摘流，不杀进程（等依赖恢复）
K8s 原语livenessProbereadinessProbe
输出颗粒度粗（活/死）细（每依赖一项 check）

真值表 — 平台状态聚合规则

L1L2聚合状态平台动作
✅✅online正常接流量
✅❌degraded不发新流量，等依赖恢复，不重启
❌❌offline触发重启（systemd/docker）
❌✅不可能探针实现 bug，排查

销售 Agent 业务场景示例

场景L1L2看板状态处置
正常运行✅✅🟢 online无需干预
Milvus 集合被误删✅❌🟡 degraded告警 → 修 Milvus，不重启 Agent
LLM Gateway 超时✅❌🟡 degraded告警 → 查 Gateway
Agent OOM killed❌❌🔴 offlinesystemd 自动重启
配置缺 SP_JWT_SECRET✅❌🟡 degraded补配置 → 重启
Agent 进程死锁❌❌🔴 offline重启
Key Insight

绝大多数 Agent 故障不是进程死了，而是依赖挂了。只做 L1 会导致"假绿"——看板一片绿、用户却一直报错，一次性击碎同事对平台的信任。L2 才是用户实际体验的 Agent 可用性。

D04 · 编排路径：复用星形拓扑 + Planner

路径技术栈复用 agent-platform推荐
A2A 消息协议（Agent 间直调）MCP / ACP over HTTP/gRPC❌ 需新写协议栈，N×N 依赖爆炸❌
独立工作流引擎Airflow / Temporal / Argo❌ 并行维护两套控制面❌
星形拓扑 + Planner（现有）Planner + MainAgentDispatcher + ExecutionCoordinator + ToolRegistry✅ 直接复用✅

星形拓扑核心约束（源自 agent-integration-spec.md §9）：

  • Sub-Agent 不直接互相通信
  • 所有跨 Agent 调用由 Main Agent / Planner 统一调度
  • 避免依赖图爆炸，便于审计和回滚

演进路径

  1. 短期：各部门 Agent 按 AGENT.yaml 接入 → AgentLoader 自动注册 → Dispatcher 独立调用。
  2. 中期：启用 MainAgentDispatcher，做跨部门 Agent 星形协同（例：销售 Main Agent 调 rag-query + presales-agent）。
  3. 长期：复杂业务流用 Planner + ExecutionCoordinator 做 DAG 编排。

D05 · 严肃业务 Human-in-the-Loop 三段式

Lesson Learned

AI-native-organization.html 已定调 Agent-as-Tool，不自治。SensePedia 回溯证明 Agent 在缺乏监督时系统性失败——缺主动思考、缺长程规划、漂移。

HITL 三段式

阶段负责平台角色
Draft 初稿 / 检索 / 分析Agent 自主执行（80% 体力活）无阻断，全量计量
Review Gate 关键决策点人类必审：发邮件 / 签合同 / 审批单 / 发布内容 / 部署生产 / 客户外发强制 Gate，未审不得出
Execute 动作执行Agent 执行已批准动作审计留痕（复用 .coord/approvals/）

为什么让同事觉得 Agent 真实有帮助

  • 同事保留最终决策权 → 心理安全 → 愿意用
  • Agent 承担 80% 体力活（写初稿 / 查资料 / 填表 / 起草方案）→ 真实省时间
  • 错了有 Gate 拦住 → 不会塌房 → 不会因一次事故永久否定 Agent
  • 复用 merge_gate / deploy_gate / rollback_gate 机制天然支持

严肃业务红线

Never — Agent 自主闭环
  • 合同签署
  • 财务付款
  • 战略决策
  • 外部邮件发送
  • 生产环境部署
Yes — AI 起草 + 人类审核
  • 内部客服话术初稿
  • 售前方案起草
  • 代码 PR 起草
  • 财务对账异常提示
  • 战略数据摘要

D06 · 平台定位：Enablement-First + Gate-on-Demand

维度控制面（Gatekeeper）服务面（Enablement） MVP 选此
权力模型有否决权，Agent 必须申请无否决权，提供能力供选择
类比K8s control plane / 合规审批 / 海关Prometheus + Service Registry / 自助加油站
典型能力Gate 审批 / 配额强制 / RBAC / kill switch注册表 / 监控看板 / 日志 / token 计量 / 共享工具库
组织含义平台团队强势 → 易和业务撕平台团队是服务商 → 易协作、易拉通
MVP Decision

先服务面，不做控制面。 Day 1 上：注册表、Token 看板、OpenClaw 状态监控、共享 Memory / RAG（纯服务面）。Phase 2 加：针对"高风险动作"（外发 / 合同 / 生产部署）按需启用 Gate。

底层逻辑：一上来就强势控制，各部门会说"我自己做"，平台就废了。先做有用、不烦人的，再逐步拿权 —— 因为信任所以简单。

§ 05Phase 1 MVP 范围与里程碑

5.1 范围边界

In Scope
  1. agent-platform 多租户化（tenant_id 贯通 + 部门 RBAC）
  2. OpenClaw 观测层量产化（4 张表 + Collector + 前端 3 页）
  3. OpenClaw Sidecar（跨网络推送链路）—— 网络拓扑决定必选：agent-platform 在机房网络，各部门 OpenClaw 在办公网络；机房 ↛ 办公（Collector 拉取不可达），办公 → 机房（Sidecar 推送可达）
  4. 1–2 个试点部门按 AGENT.yaml 接入
  5. 事业群级 Token 看板
  6. HITL Gate 基础版（schema + 审批 API，暂不强制）
Out of Scope
  • SLO / L3 质量监控
  • 跨部门 Agent 星形协同实际编排
  • 强制 Gate 拦截
  • 本地 JSONL token fallback

§ 07待议问题 · Open Questions

Q1 · 试点部门优先级

Phase 1 优先落地哪 1–2 个部门？

当前建议（按前期研究结论）：销售 + 售前 双部门，知识库飞轮抓手最明确。

需下次会议拉齐：是否同意？若不同意替代方案？部门侧是否有对接人？

Q2 · 长期协作编排的真实业务场景

"跨 Agent 编排"到底要解决哪些场景？

假设示例（需验证是否真实发生）：

  • 销售 Agent 拉客户需求 → 售前 Agent 起草方案 → 产品研发 Agent 查 POC
  • 售后 Agent 识别故障模式 → 产品研发 Agent 进 bug 列表 → 战略运营 Agent 汇总复盘
  • 财务 Agent 预警异常 → 销售 Agent 核对合同 → 战略运营 Agent 评估风险

意义：若真实场景不足 3 个，长期编排能力不建造。

§ 09与战略大图的关系

9.1 对齐 AI-Native 组织战略

  • ✅ Agent-as-Tool：平台默认 HITL，不鼓励 Agent 自主
  • ✅ 分层配给：Token 看板使 "Tier 01/02/03" 配给决策可量化
  • ✅ 飞轮抓手：销售 + 售前知识库飞轮通过共享 RAG / Memory 中台放大

9.2 对齐 Jeff 的 "Agent COE" 叙事

  • ✅ Token 可视化 = 降本量化的必要条件
  • ✅ HITL Gate = "Agent 做体力活、人类做决策" 的组织保障
  • ✅ 多租户 = 跨部门成本分摊与 ROI 核算的基础

9.3 与多 Agent 框架对比研究衔接

  • 本平台采用 CLAUDE.md 协议（TeamCreate / TaskCreate / Gate）作为平台团队内部协作规范
  • 业务部门 Agent 不强制用 CLAUDE.md 协议；只要遵循 AGENT.yaml 接入规范即可
