这里需要明确区分两个层次：Agent Router 负责在 API Gateway 之后判断当前请求应该进入哪个业务 Agent； LLM Planner 属于 SensePedia Core，负责在选中的业务 Agent 内部进行任务拆解、工具选择和执行规划。 前者解决“选哪个 Agent”，后者解决“选中后怎么执行”。

在工具接入层，平台保留 Tool Registry + Skills + HostOps API 作为原生能力，同时支持通过 MCP Adapter + MCP Client 接入外部 MCP Server。其中 MCP Client 负责按协议连接和调用， MCP Adapter 负责把 MCP 能力映射为平台内部统一工具模型。MCP 的定位是标准化外部工具协议， 不是替代 SensePedia Core、Knowledge Infrastructure 或 HostOps API 的内部执行内核。

另外，Checkpoint Store 与 Memory Service 不是替代关系。 Checkpoint Store 保存的是一次执行过程中的短期运行态与断点恢复快照；Memory Service 保存的是跨会话、可召回的长期记忆与 compaction 写回结果。 前者解决执行连续性，后者解决知识连续性。

Multi-Agent 协作模式（Phase 2）

Multi-Agent 在这里被定义为 可选扩展能力，不是所有项目的默认执行模式。MVP 阶段仍然建议优先采用 single agent + shared tools；只有当一个业务请求必须拆给多个专业 Agent 协同时，才启用 Main Agent 协调模式。

第一版推荐采用星型拓扑：Main Agent 作为唯一协调者，负责分派、聚合和冲突裁决； Sub Agent 不直接互相通信，只通过 Main Agent 回传结果。这样可以保证 trace、权限、审计和 retry 逻辑都留在单一控制面。

推荐协作协议
interface SubAgentCall {
  call_id: string;
  target_agent: string;
  goal: string;
  input_payload: Record<string, unknown>;
  allowed_tools?: string[];
  timeout_ms: number;
  stream: boolean;
}

interface SubAgentResult {
  call_id: string;
  target_agent: string;
  status: "success" | "partial" | "failed";
  confidence?: number;
  evidence_refs?: string[];
  output_payload: Record<string, unknown>;
  events?: Array<"agent_started" | "partial_result" | "agent_completed">;
}
默认治理规则
1. 通信边界：Sub Agent 之间不直连，避免形成不可解释的 agent-to-agent 黑盒。
2. 结果聚合：Result Aggregator 负责去重、字段合并、证据拼接，并可按事件流输出阶段性结果。
3. 冲突裁决：优先看领域优先级，其次看证据新鲜度与 confidence；仍无法决策时进入人工复核或返回带保留意见的答复。
4. 典型场景：例如 Legal Agent 主控下并行调用 RAG Agent 做法条检索、OCR Agent 做材料识别，再由主 Agent 统一生成最终法律结论。

系统稳定性保障 - 降级方案

Agent 模式会多次调用外部服务（LLM、检索服务），存在QPS 放大效应，需要完善的降级方案保障系统稳定性。

→

→

→

→

LLM 多 API 降级
SAAS API 场景：配置多个 API Key（如主用 Claude，备用 GPT-4），主节点故障时自动切换到备选 API。
本地部署场景：建议只做服务不可用时的快速失败，不做 API 级别降级。
QPS 限流
全局限流：限制系统总并发请求数，防止过载。
单用户限流：限制单个用户的并发请求数，防止资源抢占。
Agent 放大控制：限制 Agent 单次任务的最大工具调用次数。
熔断机制
超时熔断：LLM 调用、检索服务响应超时（建议 30s）自动放弃，避免请求堆积。
错误熔断：连续失败 N 次后熔断该服务，恢复后逐步放行流量。
半开状态：熔断恢复时先放行少量请求探测，可用则恢复正常。
服务降级策略
检索不可用：返回"知识库暂不可用，请稍后重试"，不返回错误信息。
LLM 不可用：返回"服务繁忙，请稍后重试"。
Agent 任务超时：保留已执行结果，返回部分答案并提示"答案未完整"。

实现位置：熔断限流模块位于 API Gateway 与 LLM Gateway 之间，统一处理限流、熔断、超时控制。

Python 源码加密方案（客户私有化部署 IP 保护）
为保障客户私有化部署时的知识产权保护，需要对 Python 源码进行加密处理。采用 Cython 方案将源码编译为二进制扩展模块（.so 文件），使逆向工程难度大幅提升。
加密原理
Python (.py) → C 代码 (.c) → 编译为 C 扩展模块 (.so/.pyd)，编译后的二进制代码难以直接阅读和逆向。
入口文件保留
启动脚本、配置文件、__init__.py 等入口文件保持 .py 格式以便修改；业务逻辑代码全部编译为 .so。
不兼容文件处理
部分 Python 特性（如 redis 向量操作）无法编译，需保持 .py 源码或使用替代方案。
编译脚本
使用 setup_main.py 自动遍历项目目录，调用 setup.py 逐文件编译，并打包至 package 目录交付。
部署包结构
package/
├── app/                    # 编译后的 .so 模块
│   └── __init__.py        # 入口文件（保持 .py）
├── skills/                 # 编译后的 skill 模块
├── agent_platform/         # 编译后的核心模块
├── setup.py               # 编译入口（交付用）
├── setup_main.py          # 自动编译脚本（仅开发使用）
├── requirements.txt       # 依赖清单
└── Dockerfile            # 容器构建文件
编译流程
  1. 开发环境安装 Cython: pip install Cython
  2. 执行 python setup_main.py 自动编译
  3. 脚本自动添加 # cython: language_level=3 头部
  4. 调用 setup.py 编译为 .so 文件
  5. 移动 .so 至 package 目录
  6. 复制非编译文件（配置、入口、__init__.py）
  7. 交付 package 目录给客户

注意事项：编译前需在目标运行环境中测试，因为不同 Python 版本、操作系统编译的 .so 文件可能存在兼容性问题。建议在 Docker 容器中统一编译环境。

业务 Agent 扩展设计

平台采用插件化架构，通过定义通用的 Agent 基类和标准接口，可以快速接入新的业务 Agent。每个 Agent 只需实现特定的业务逻辑，通用能力由平台提供。

这些业务 Agent 是 Agent Router 的候选目标。Router 会基于请求意图、租户配置、权限范围和可用工具，选出一个 Selected Business Agent，随后再进入该 Agent 内部的 LLM Planner 与 Graph 执行流程。

BaseBusinessAgent 契约（Template Method + Shared Services）

为了避免 8 个 Business Agent 重复实现 Memory 加载、权限校验、Guardrail 和输出包装，推荐由 BaseBusinessAgent 固定执行六段式骨架流程，业务 Agent 只通过 hook 覆盖领域差异。 通用能力应尽量下沉到共享服务，而不是散落在每个 Agent 的自定义代码里。 

class BaseBusinessAgent:
    def handle(self, request: AgentRequest) -> AgentResponse:
        normalized = self.request_adapter(request)
        context = self.context_memory_adapter(normalized)
        plan_input = self.build_plan_input(normalized, context)
        plan = self.plan_and_validate(plan_input)
        execution = self.skill_orchestrator(plan, context)
        checked = self.guardrail(execution, context)
        return self.response_adapter(checked, context)

    # business hooks
    def manifest(self) -> AgentManifest: ...
    def allowed_tools(self) -> list[str]: ...
    def response_schema(self) -> type[BaseModel]: ...
    def domain_prompt(self) -> str: ...
    def post_process(self, result: AgentResult) -> AgentResult: ...
Hook / Contract 职责 默认实现 / 约束
manifest() 声明 agent_id、领域描述、默认路由标签、审批等级 Agent Registry 读取 manifest，Router 只面向标准契约
allowed_tools() 定义当前业务 Agent 可见的工具白名单与并发预算 Tool Registry 与 PlanValidator 共同执行越权拦截
domain_prompt() 注入领域目标、推理边界、答复风格与引用要求 只扩展 planner 输入，不改六步主流程
response_schema() 定义最终输出结构，例如 citation-heavy answer、OCR JSON、legal memo 由 Response Adapter 做统一 schema 校验与错误包装
post_process() 补充领域后处理，例如引用规范化、免责声明插入、格式润色 必须是纯函数式后处理，不应重新触发工具执行
AgentContext / AgentResult 统一在各阶段之间传递用户、租户、memory、tool trace、guardrail 结果 所有共享中间态都进入标准对象，避免每个 Agent 自定义字段漂移

Legal Agent 抽象设计

这类法律咨询 Agent 不应只抽象成“法条检索”或“案例摘要”，更合理的抽象是： 先完成 意图识别 + 统一结构化抽取 + 槽位门控，再进入 规则引擎适配 + 法律知识图谱检索 + 可解释答复生成。

端到端推理链路
1. Multi-Intent Classifier
对用户问题做多标签法律意图识别，例如监护、领养、父母身份、跨境拐带儿童等。
2. Unified Schema Extractor
使用统一 schema 抽取事实槽位，避免每个意图各自维护一套输入结构。
3. Slot Completeness Gate
按 critical / secondary 槽位完整性决定进入 `ASK_MORE`、`PRELIMINARY` 或 `FULL_ADVICE`。
4. Rule Adapter Layer
若信息足够，则根据领域规则将统一结构映射到对应规则引擎或法条章节适配层。
5. Legal Knowledge Graph Retriever
从法律知识图谱或法条知识库中检索相关法条原文、条款关系和来源标识，作为答复依据。
6. Answer Composer
生成自然语言答复，并固定输出 assumptions、缺失事实和 disclaimer，保证可解释与合规。
可复用模块抽象
Intent Layer
多意图分类器，支持单标签与多标签并存，适合法律、合规、医疗等高门槛咨询场景。
Fact Layer
统一结构化抽取器 + 槽位完整性校验器，负责把自然语言转成规则引擎可消费的结构。
Decision Layer
Response Mode Controller 统一决定 `ASK_MORE / PRELIMINARY / FULL_ADVICE`，避免直接进入高风险推理。
Reasoning Layer
Rule Adapter Layer 负责“追问优先、skip 兜底、默认值可解释”，避免事实缺失时误推理。
Evidence Layer
Legal Knowledge Graph Retriever 负责法条、知识图谱节点、来源章节与关系边的检索与返回。
Compliance Layer
Assumptions / Missing Facts / Disclaimer Composer 统一负责对用户展示不确定性、缺失项与结论边界。
ASK_MORE / PRELIMINARY / FULL_ADVICE 的抽象意义
ASK_MORE
critical 槽位缺失时，系统只追问，不给出正式法律结论。
PRELIMINARY
secondary 槽位缺失时，列出缺失项，给初步分析，但明确非最终结论。
FULL_ADVICE
信息完整、规则依据充分时，输出正式分析，同时附免责声明与依据来源。
当前边界与后续扩展
当前主依据
当前回答以规则引擎结论 + 法律知识图谱 / 法条原文检索为主，不依赖判例参考作为主链路。
可解释性约束
每个结论都应绑定 source norm id / chapter，并限制 LLM 仅使用检索到的法律上下文。
后续扩展
可以逐步加入判例参考、跨法域适配、知识图谱自动更新和更复杂的审查工作流。
知识库创建与检索流水线

知识库能力分成两条主线：Knowledge Ingestion Agent 负责文档入库与发布， RAG Agent 负责查询时的 Hybrid Search 检索与答案生成。除此之外，Knowledge Ingestion Agent 还需要负责网页类知识源的定时监控、变更检测和增量重发。

入库流水线（Knowledge Ingestion Agent）
1. 上传登记
用户上传原始文档后，原始文件写入 MinIO，同时在 MariaDB 中记录文档元数据、状态和更新时间。
2. Markdown 转换
pdf/doc/docx/ppt/pptx 通过 DocumentParser 异步转 Markdown；txt/md/xlsx 走本地转换逻辑。
3. 解析产物落盘
生成的 Markdown、图片、附件等资源写回 MinIO，并把资源关系和文档状态写入 MariaDB。
4. Parent Chunk 阶段
调用 splitter 逻辑切 Parent Chunk，带 chunk 标记的 Markdown 上传 MinIO，中间结果写入 MongoDB。
5. Sub Chunk + 向量生成
继续切成最终检索用的 Sub Chunk；通过 llm_server 生成 Dense Embedding，并在应用侧训练 BM25 编码器生成 Sparse Embedding。
6. 发布入库
RabbitMQ 串联各异步阶段，将 Sub Chunk 的 dense_embedding 与 sparse_embedding 一起写入 Milvus，文档随后进入可检索状态。
关键点：真正写入 Milvus 的是 Sub Chunk 的稠密/稀疏向量；BM25 训练与 sparse_embedding 生成发生在应用侧，不发生在 Milvus 内部。
检索流水线（RAG Agent）
1. Dense Query
查询文本通过 llm_server `/v1/embeddings` 生成查询稠密向量，在 Milvus 中执行 Dense Search。
2. Sparse Query
查询文本通过对应文档 BM25 编码器生成 query sparse vector，在 Milvus 中执行 Sparse Search。
3. Hybrid Fusion
Milvus 对 Dense 与 Sparse 结果做 Hybrid Search，并通过 RRF 进行融合排序。
4. Answer Composition
RAG Agent 对检索证据进行聚合、引用归一化和回答生成，输出带引用的最终答案。
一句话概括：知识库创建本质上是 RabbitMQ + MariaDB + MongoDB + MinIO + llm_server + BM25 本地编码器 + Milvus 共同协作完成的异步入库链路。
网页知识源自动更新链路（Knowledge Ingestion Agent）
1. 定时抓取
对配置过的网页知识源执行定时爬虫任务，例如组织架构、政策公告、规则说明页等。
2. 变更检测
将新抓取页面与上次快照对比；如果正文、结构或关键字段没有变化，则跳过，不重复入库。
3. 自动判定
若变化较小且风险低，可直接触发增量更新；若变化较大、解析不稳定或涉及关键知识页，则转入人工确认。
4. 收件箱审批
向系统页面的收件箱推送待处理事项，界面显示红点提醒；运营人员进入收件箱查看 diff、来源页面和建议操作后决定是否发布。
5. 重新入库发布
人工确认或自动放行后，Knowledge Ingestion Agent 重新走 Markdown 转换、切分、向量化与 Milvus 发布流程，替换旧版本知识。
推荐策略：把“自动检测”与“人工审批”做成可配置模式。低风险网页走自动更新，高风险或重要网页走 Inbox + Human-in-the-loop。
自动更新能力的系统落点
Knowledge Ingestion Agent
负责抓取、比对、决定是否触发重新入库，并调用知识库更新接口。
Web Crawler / Change Detector
负责定时抓取网页、生成页面快照与结构化 diff。
Ops Inbox / Approval Queue
负责把待审批更新推送到系统页面，支持红点提醒、人工确认与操作留痕。
Knowledge Infrastructure
复用现有 MinIO、MariaDB、MongoDB、RabbitMQ、llm_server、BM25 与 Milvus，不需要单独再造一条入库底座。
这意味着 Knowledge Ingestion Agent 不再只是“文档上传入库 Agent”，而是一个同时覆盖 人工上传 + 网页源同步 + 增量重发 的知识库生命周期 Agent。

为什么单独做 Memory Service
不是业务 Agent
Memory 的目标是为 Agent 提供长期记忆写入、整理、召回与压缩，不是直接面向终端用户独立回答问题。
复用入库原语
从 Knowledge Ingestion Agent 复用 normalize、chunk、embedding、sparse encode、publish、source monitor、审批回路这些底层能力。
复用检索原语
从 RAG Agent 复用 query embedding、hybrid retrieval、rerank、top-k merge、作用域过滤，但不复用完整 citation-heavy answer 流程。
保留独立生命周期
Memory 有自己的 daily memory、MEMORY.md、temporal decay、compaction writeback、dirty sync 和 graceful degradation，不应和知识库文档生命周期混为一体。
Memory Service 内部能力
Truth Source
`memory/roles/*` + `memory/users/*` + `memory/bindings/*`，Markdown 文件即真相源。
Write Path
先按 scope 归属路由：角色固有设定写入 role scope，用户稳定偏好与长期事实写入 user scope，当前角色下形成的专属互动与关系记忆写入 user-role scope；随后统一进入 Memory Writer，落 Markdown 真相源并触发索引更新与 compaction。
Recall Path
按三层 scope 优先级召回：先 user-role，再 user，最后 role；完成 scoped recall 后再做 temporal decay、hybrid merge、去重与 prompt injection，避免高专属性记忆被低专属性基线记忆淹没，不直接强耦合 RAG 回答器。
Sync Path
watcher + dirty flag + interval sync + graceful fallback，保证 memory 始终是活数据索引；底层可复用 MariaDB / Milvus / MinIO，但 memory collection、metadata table、truth-source 路径与知识库分域隔离。
建议的复用边界
直接复用的能力
Chunker、Embedding Pipeline、BM25 Encoder、Hybrid Search、Reranker、Version Replace、Web Diff、Inbox 审批。
不要直接复用的整链路
普通 memory 写入不应默认走完整知识库重型入库链；memory recall 也不应默认走完整 RAG answer composition。
Compaction 的落点
Compaction 产生 durable memory 候选后，进入 Memory Writer，写回 Markdown truth source，再触发增量索引。
平台中的位置
Memory Service 应该位于支撑系统层，与 Knowledge Infrastructure 共用基础设施，但采用分开数据域的方式管理 memory metadata、memory index 和 memory truth source，并服务于 SensePedia Core 与各业务 Agent。
Self-Improving Agent 设计
LangGraph 负责编排，不会天然“自我进化”。平台需要额外构建一个反馈学习闭环，把用户显式反馈、隐式行为信号与执行轨迹沉淀为可验证的策略更新， 优先优化 Prompt、Tool Routing、Retrieval 参数与用户偏好记忆，而不是直接在线改模型权重。
1. Feedback Capture
从前端、API 和业务日志采集显式反馈与隐式反馈，形成统一反馈事件流。
  • 显式反馈：点赞、点踩、纠错、人工改写、满意度评分
  • 隐式反馈：是否继续追问、是否复制答案、是否采纳结果、任务是否完成
  • 上下文：用户 ID、线程 ID、Prompt 版本、工具链路、引用与耗时
2. Evaluation & Attribution
用 evaluator/critic 节点和离线分析，对问题根因进行归因，而不是把所有负反馈都归因给模型。
  • 检索不足：召回文档不对、topk 不够、过滤条件错误
  • 工具决策失误：选错 Tool、参数映射错误、执行顺序不合理
  • 答案质量不足：结构混乱、未结合证据、缺少引用或引用错误
3. Policy Update
将归因结果转为低风险、可回滚的策略更新，写入平台配置中心和长期记忆。
  • 更新 Prompt 模板、Few-shot 示例和输出 schema
  • 调整 Tool 优先级、Planner 路由规则、并行执行策略
  • 调节 Retrieval 参数、重排阈值、用户偏好配置
4. Safe Rollout
新策略必须先经过离线回放与灰度验证，再逐步放量，避免自我进化变成自我退化。
  • 历史样本回放、自动评测、人工抽检
  • 按租户/用户分桶灰度发布，比较成功率与 fallback rate
  • 策略版本化与一键回滚

维护可生效的 Prompt、Router、Retrieval、Memory 策略版本。
Experiment Manager
支持 A/B、canary、tenant-level rollout 和效果对比。
安全护栏
只允许更新“可解释、可回滚”的配置层，不允许 agent 直接在线修改核心代码和模型权重。
任何策略更新都必须绑定评估指标，如 grounded answer rate、tool error rate、citation coverage、user success rate。
将高风险领域的更新限制为“建议模式”，需要人工审批后才能进入生产。
用户隔离必须保留，单个租户或用户的反馈不能污染其他租户的默认策略。
Agent Router 分层与可插拔设计

为了同时支持单需求单 Agent和多需求多 Agent两种交付模式，Agent Router 保持统一入口与统一输出契约， 但内部实现拆分为可替换的策略模块。这样既不破坏总链路，也避免在单 Agent 项目里强行启用复杂的多 Agent 智能路由。

对外仍然是 Gateway 之后进入 Router、产出 Selected Business Agent；对内则拆为 Registry、Policy、Strategy、Dispatcher 四层， 通过配置切换运行模式，而不是修改主流程代码。

1. 稳定契约层

Router 对上游只暴露统一的 route(request) 接口，对下游只产出统一的 RouteDecision，包含 selected_agent、reason、policy_result、route_mode。

2. 可替换策略层
  • single_agent: 固定返回唯一 Agent，例如 legal_agent
  • configured_routing: 按租户、渠道、权限、功能标签静态分流
  • dynamic_routing: 按规则 + 意图识别 + fallback 动态选 Agent
3. 可插拔扩展点
  • AgentRegistry: 基于 manifest.yaml 自动发现 Agent
  • RoutePolicy: 做租户、权限、灰度、黑白名单过滤
  • RoutingStrategy: 负责真正的 Agent 选择逻辑
  • Dispatcher: 只负责把已选中的上下文传给业务 Agent
Request
  -> AgentRegistry(list available agents for tenant/channel)
  -> RoutePolicy(filter by auth / role / tenant / allowlist)
  -> RoutingStrategy(select one agent from allowed candidates)
  -> RouteDecision(selected_agent, reason, route_mode, policy_result)
  -> Dispatcher(forward request + routing context)
  -> Selected Business Agent
运行模式 适用场景 Router 行为 是否需要复杂意图识别
single_agent 单企业、单需求、单 Agent 交付 静态绑定唯一 Agent，只保留准入校验、审计、灰度开关 否
configured_routing 少量需求、少量 Agent、规则边界清晰 按 tenant / channel / capability / request_type 进行规则路由 通常不需要
dynamic_routing 平台化、多需求、多 Agent 协同 基于规则、优先级、意图判断、fallback 选出最合适 Agent 是，可按需启用 
router:
  mode: single_agent            # single_agent | configured_routing | dynamic_routing
  default_agent: legal_agent
  strategy: fixed_agent         # fixed_agent | rule_based | intent_based | hybrid
  enable_policy_filter: true
  enable_fallback: true
这意味着 Router 的职责边界不变，仍然只负责“选哪个业务 Agent”；变化的是其内部实现从单一方案改为分层 + 可插拔 + 可退化。 单 Agent 项目使用最薄的固定路由策略，多 Agent 平台再启用规则路由或动态路由能力。
LLM Planner 详细设计

本节描述的是 SensePedia Core 中的 LLM Planner 如何工作。它发生在选中某个业务 Agent 之后， 不负责业务 Agent 的选择；业务 Agent 选择由上游的 Agent Router 负责，LLM Planner 只负责当前 Agent 的内部执行规划。

Planner 核心功能
1. Task Analyzer (任务分析)
• 分析用户意图
• 识别任务类型
• 提取关键实体
2. Plan Generator (计划生成)
• 分解复杂任务为子任务
• 确定执行顺序
• 识别并行执行机会
3. ReAct Reasoning (推理行动)
• 思考 (Thought)
• 行动 (Action)
• 观察 (Observation)
4. Tool Selector (工具选择)
• 根据任务选择合适工具
• 参数映射
• 依赖分析

LLM Planner 与其他模块的交互关系
交互模块 交互方向 交互内容
Agent Router ─ 输入边界 Agent Router 先选定当前请求所属的业务 Agent，LLM Planner 不参与“选哪个 Agent”，只接收已选中的执行上下文。
Selected Business Agent ─ 所属上下文 LLM Planner 运行在当前被选中的业务 Agent 内部，继承该 Agent 的领域目标、工具范围与执行约束。
Tool Registry ─ 双向 查询可用工具列表、获取工具 schema、验证工具存在性，并统一路由到 Local Skills、HostOps API 或 MCP Tools
MCP Adapter ─ 间接 通过 Tool Registry 间接消费 MCP Client 拉取到的能力，将 tools/resources/prompts 映射为平台统一 schema，并注入权限、审计、trace 与租户上下文。
MCP Client ─ 间接 负责与外部 MCP Server 建立连接、初始化 session、拉取 capabilities，并按 MCP 协议转发工具调用。
LLM Gateway ─ 输出 发送 structured output 请求、接收符合 ExecutionPlan schema 的计划候选
PlanValidator ─ 双向 对 Planner 产物做 schema、工具白名单、依赖 DAG、并发预算与参数合法性校验，失败时把错误原因回传给 Planner 修复
Graph Executor ─ 输出 接收 validated plan、执行任务图并在失败时触发 replan
Local Skills ─ 间接 通过 Tool Registry 间接调用平台内置 Skill；与 MCP Tools 并列作为两类工具来源。
Knowledge Infrastructure ─ 间接 在知识库相关 Agent 中，LLM Planner 会围绕检索、入库、网页更新等任务规划对共享知识基础设施的调用顺序。
Memory Service ─ 双向 读取长期记忆与偏好上下文，并在需要时把新的 durable memory、摘要或 compaction 写回候选交给 Memory Service。
HostOps API ─ 双向 当任务涉及宿主机、容器、进程、文件系统或远程服务器操作时，通过统一 HostOps API 发起执行请求，并接收审批状态、执行结果与审计信息。
Observability ─ 输出 记录规划过程、耗时、决策依据
LLM Planner 结构化输出与验证门

不建议让 Planner 直接输出“自由 JSON”。更稳妥的做法是让模型按固定 schema 生成 structured output，随后进入独立的 PlanValidator 节点做执行前校验。 这样模型负责“规划意图”，平台负责“计划合法性”。 

class PlanTask(BaseModel):
    id: str
    tool: str
    params: dict[str, Any]
    depends_on: list[str] = []
    parallel_group: str | None = None

class ExecutionPlan(BaseModel):
    reasoning_summary: str
    agent_scope: str
    tasks: list[PlanTask]

planner = planner_llm.with_structured_output(ExecutionPlan)
plan = planner.invoke(planner_input)
validated_plan = PlanValidator.validate(
    plan=plan,
    allowed_tools=current_agent.allowed_tools(),
    tool_registry=tool_registry,
    max_tasks=8,
)
PlanValidator 至少检查五件事：工具是否存在、参数是否满足 schema、是否越过当前 Agent 的工具边界、 depends_on 是否形成合法 DAG、任务数与并行度是否超预算。校验失败时，Planner 根据错误原因重试修复，而不是把非法计划直接交给 Executor。 
你是 SensePedia Core 中的 LLM Planner。

你不负责选择业务 Agent。
当前请求已经经过 Agent Router，并且已经确定了 Selected Business Agent。
你的职责是在当前业务 Agent 的上下文内完成任务拆解、工具选择与执行规划。
你必须严格按照 ExecutionPlan schema 返回结构化结果，不要输出自由文本 JSON。

MCP 协议接入设计

平台基于 Python LangGraph 构建，在保留内置 Skills、Knowledge Infrastructure 与 HostOps API 的基础上， 支持通过 MCP（Model Context Protocol） 接入外部工具、资源与提示模板。MCP 在本架构中的定位是 标准化外部工具接入协议，而不是替代平台内部执行内核。

接入链路 
LLM Planner
  -> Tool Registry
  -> MCP Adapter
  -> MCP Client
  -> External MCP Servers
  -> tools / resources / prompts
设计原则
1. 统一入口：Planner 仍只面向 Tool Registry 选工具，不直接绑定具体 MCP Server。
2. 协议角色：MCP Client 只负责按 MCP 协议连接 server、管理 session、拉取 capabilities 与转发调用。
3. 平台适配：MCP Adapter 负责 schema 映射、权限注入、错误归一、trace 透传与 Tool Registry 集成。
4. 边界清晰：知识库、长期记忆与高治理 HostOps 能力默认优先走平台原生接口；第三方工具生态优先通过 MCP 接入。
5. 并列来源：Local Skills 与 MCP Tools 并列存在，由 Tool Registry 统一暴露给 LangGraph 执行流。
MCP Client / Adapter 建议职责 
interface MCPClientSession {

在实现上可以把这两层先写在同一个进程里，但在架构表达上建议继续拆开： MCP Client 是协议角色，MCP Adapter 是平台适配层。

如果后续需要把平台能力向外暴露为标准协议，也可以在此基础上继续扩展为平台侧 MCP Server； 但当前推荐优先落地的是 “LangGraph Agent 作为 MCP Client 消费外部 MCP Server” 的模式。

HostOps Runtime 分层方案（Python LangGraph + HostOps API + Pi/自研 Runtime）

当平台需要具备类似 OpenClaw + Pi 的宿主机/服务器操作能力时，最稳妥的做法不是把宿主机执行逻辑直接塞进业务 Agent， 而是在 SensePedia Core / Python LangGraph 之外新增一层独立的 HostOps API， 由它统一承接权限校验、审批挂起、审计留痕、错误模型与能力发现，再把真正执行下沉到 Pi 或 自研 TS Runtime。

这样分层的核心价值是：业务编排层继续专注“怎么思考和拆任务”， 执行层专注“怎么安全地碰宿主机和服务器”。

各层职责
1. Python LangGraph（业务编排层）
• 任务分析 / 路由 / 规划
• 工具选择与调用顺序
• 不直接持有宿主机权限
2. HostOps API（能力抽象层）
• 统一 `exec / ssh / files / process / docker / browser` 契约
• 审批、审计、权限、配额、租户隔离
• 将执行层差异对上层隐藏
3. Pi / 自研 TS Runtime（执行层）
• 真正执行命令、进程、文件、远程连接
• 对接 sandbox / session / process 生命周期
• 不承担上层业务 Agent 的推理职责

Runtime 选型对比
维度 自研 TS Runtime 直接使用 Pi
上线速度 慢，前期研发成本高 快，适合先做 MVP
架构控制力 最高，接口、状态、权限都可自定义 中等，需要适配 Pi 的模型
与现有系统契合度 最高，能贴合审批流、审计、租户隔离 中等，需要桥接层
长期策略 长期最优 短期最优
推荐用法 平台化、私有化、企业级强治理场景 快速验证、早期试点、先跑通闭环
Temporal 的建议落点
Temporal 不是 Agent Brain，而是可选的 durable workflow backend。 当 HostOps 进入长任务、跨机器步骤编排、审批后恢复、失败重试、定时等待等场景时， 可以在 HostOps API 之下把 Temporal 作为可靠执行底座；而在 MVP 阶段通常可以先不引入， 先以 Pi 或自研 TS Runtime 跑通最小闭环。
