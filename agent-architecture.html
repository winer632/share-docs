这里需要明确区分两个层次：Agent Router 负责在 API Gateway 之后判断当前请求应该进入哪个业务 Agent； LLM Planner 属于 SensePedia Core，负责在选中的业务 Agent 内部进行任务拆解、工具选择和执行规划。 前者解决“选哪个 Agent”，后者解决“选中后怎么执行”。

另外，Checkpoint Store 与 Memory Service 不是替代关系。 Checkpoint Store 保存的是一次执行过程中的短期运行态与断点恢复快照；Memory Service 保存的是跨会话、可召回的长期记忆与 compaction 写回结果。 前者解决执行连续性，后者解决知识连续性。

Legal Agent 抽象设计

这类法律咨询 Agent 不应只抽象成“法条检索”或“案例摘要”，更合理的抽象是： 先完成 意图识别 + 统一结构化抽取 + 槽位门控，再进入 规则引擎适配 + 法律知识图谱检索 + 可解释答复生成。

端到端推理链路
1. Multi-Intent Classifier
对用户问题做多标签法律意图识别，例如监护、领养、父母身份、跨境拐带儿童等。
2. Unified Schema Extractor
使用统一 schema 抽取事实槽位，避免每个意图各自维护一套输入结构。
3. Slot Completeness Gate
按 critical / secondary 槽位完整性决定进入 `ASK_MORE`、`PRELIMINARY` 或 `FULL_ADVICE`。
4. Rule Adapter Layer
若信息足够，则根据领域规则将统一结构映射到对应规则引擎或法条章节适配层。
5. Legal Knowledge Graph Retriever
从法律知识图谱或法条知识库中检索相关法条原文、条款关系和来源标识，作为答复依据。
6. Answer Composer
生成自然语言答复，并固定输出 assumptions、缺失事实和 disclaimer，保证可解释与合规。
可复用模块抽象
Intent Layer
多意图分类器，支持单标签与多标签并存，适合法律、合规、医疗等高门槛咨询场景。
Fact Layer
统一结构化抽取器 + 槽位完整性校验器，负责把自然语言转成规则引擎可消费的结构。
Decision Layer
Response Mode Controller 统一决定 `ASK_MORE / PRELIMINARY / FULL_ADVICE`，避免直接进入高风险推理。
Reasoning Layer
Rule Adapter Layer 负责“追问优先、skip 兜底、默认值可解释”，避免事实缺失时误推理。
Evidence Layer
Legal Knowledge Graph Retriever 负责法条、知识图谱节点、来源章节与关系边的检索与返回。
Compliance Layer
Assumptions / Missing Facts / Disclaimer Composer 统一负责对用户展示不确定性、缺失项与结论边界。
ASK_MORE / PRELIMINARY / FULL_ADVICE 的抽象意义
ASK_MORE
critical 槽位缺失时，系统只追问，不给出正式法律结论。
PRELIMINARY
secondary 槽位缺失时，列出缺失项，给初步分析，但明确非最终结论。
FULL_ADVICE
信息完整、规则依据充分时，输出正式分析，同时附免责声明与依据来源。
当前边界与后续扩展
当前主依据
当前回答以规则引擎结论 + 法律知识图谱 / 法条原文检索为主，不依赖判例参考作为主链路。
可解释性约束
每个结论都应绑定 source norm id / chapter，并限制 LLM 仅使用检索到的法律上下文。
后续扩展
可以逐步加入判例参考、跨法域适配、知识图谱自动更新和更复杂的审查工作流。
知识库创建与检索流水线

知识库能力分成两条主线：Knowledge Ingestion Agent 负责文档入库与发布， RAG Agent 负责查询时的 Hybrid Search 检索与答案生成。除此之外，Knowledge Ingestion Agent 还需要负责网页类知识源的定时监控、变更检测和增量重发。

入库流水线（Knowledge Ingestion Agent）
1. 上传登记
用户上传原始文档后，原始文件写入 MinIO，同时在 MariaDB 中记录文档元数据、状态和更新时间。
2. Markdown 转换
pdf/doc/docx/ppt/pptx 通过 DocumentParser 异步转 Markdown；txt/md/xlsx 走本地转换逻辑。
3. 解析产物落盘
生成的 Markdown、图片、附件等资源写回 MinIO，并把资源关系和文档状态写入 MariaDB。
4. Parent Chunk 阶段
调用 splitter 逻辑切 Parent Chunk，带 chunk 标记的 Markdown 上传 MinIO，中间结果写入 MongoDB。
5. Sub Chunk + 向量生成
继续切成最终检索用的 Sub Chunk；通过 llm_server 生成 Dense Embedding，并在应用侧训练 BM25 编码器生成 Sparse Embedding。
6. 发布入库
RabbitMQ 串联各异步阶段，将 Sub Chunk 的 dense_embedding 与 sparse_embedding 一起写入 Milvus，文档随后进入可检索状态。
关键点：真正写入 Milvus 的是 Sub Chunk 的稠密/稀疏向量；BM25 训练与 sparse_embedding 生成发生在应用侧，不发生在 Milvus 内部。
检索流水线（RAG Agent）
1. Dense Query
查询文本通过 llm_server `/v1/embeddings` 生成查询稠密向量，在 Milvus 中执行 Dense Search。
2. Sparse Query
查询文本通过对应文档 BM25 编码器生成 query sparse vector，在 Milvus 中执行 Sparse Search。
3. Hybrid Fusion
Milvus 对 Dense 与 Sparse 结果做 Hybrid Search，并通过 RRF 进行融合排序。
4. Answer Composition
RAG Agent 对检索证据进行聚合、引用归一化和回答生成，输出带引用的最终答案。
一句话概括：知识库创建本质上是 RabbitMQ + MariaDB + MongoDB + MinIO + llm_server + BM25 本地编码器 + Milvus 共同协作完成的异步入库链路。
网页知识源自动更新链路（Knowledge Ingestion Agent）
1. 定时抓取
对配置过的网页知识源执行定时爬虫任务，例如组织架构、政策公告、规则说明页等。
2. 变更检测
将新抓取页面与上次快照对比；如果正文、结构或关键字段没有变化，则跳过，不重复入库。
3. 自动判定
若变化较小且风险低，可直接触发增量更新；若变化较大、解析不稳定或涉及关键知识页，则转入人工确认。
4. 收件箱审批
向系统页面的收件箱推送待处理事项，界面显示红点提醒；运营人员进入收件箱查看 diff、来源页面和建议操作后决定是否发布。
5. 重新入库发布
人工确认或自动放行后，Knowledge Ingestion Agent 重新走 Markdown 转换、切分、向量化与 Milvus 发布流程，替换旧版本知识。
推荐策略：把“自动检测”与“人工审批”做成可配置模式。低风险网页走自动更新，高风险或重要网页走 Inbox + Human-in-the-loop。
自动更新能力的系统落点
Knowledge Ingestion Agent
负责抓取、比对、决定是否触发重新入库，并调用知识库更新接口。
Web Crawler / Change Detector
负责定时抓取网页、生成页面快照与结构化 diff。
Ops Inbox / Approval Queue
负责把待审批更新推送到系统页面，支持红点提醒、人工确认与操作留痕。
Knowledge Infrastructure
复用现有 MinIO、MariaDB、MongoDB、RabbitMQ、llm_server、BM25 与 Milvus，不需要单独再造一条入库底座。
这意味着 Knowledge Ingestion Agent 不再只是“文档上传入库 Agent”，而是一个同时覆盖 人工上传 + 网页源同步 + 增量重发 的知识库生命周期 Agent。
用户个性化记忆系统设计

这里的长期记忆由 Memory Service 承担；而总览图中的 Checkpoint Store 仍然保留，用于保存 workflow 的短期运行态与断点恢复快照。 两者分别服务于不同层级：Checkpoint 是执行态，Memory 是知识态。

短期记忆 (Working Memory)
当前会话内的上下文信息，会话结束后自动清除。类似于人类的工作记忆。
✓ 当前对话历史
✓ 本次检索的知识
✓ 临时变量/状态
✓ Graph 执行状态
实现：InMemoryCheckpointStore + State
长期记忆 (Long-term Memory)
跨会话持久化的用户偏好、学习到的信息、历史交互摘要。默认采用类似 OpenClaw 的内置长期记忆，为每个用户维护独立的长期记忆文件与检索索引。
✓ 用户偏好设置
✓ 交互历史摘要
✓ 角色/人格设定
✓ 用户专属 MEMORY.md / daily memory
✓ 自定义知识
默认实现：内置文件记忆 + 向量索引；增强选项：PostgreSQL / MemOS / Recall
共享记忆 (Shared Memory)
团队/组织共享的知识、文档、配置，团队内所有成员可见。
✓ 团队知识库
✓ 常用模板
✓ 团队规范
✓ 项目文档
实现：Team Scope 知识库 + MinIO

为什么单独做 Memory Service
不是业务 Agent
Memory 的目标是为 Agent 提供长期记忆写入、整理、召回与压缩，不是直接面向终端用户独立回答问题。
复用入库原语
从 Knowledge Ingestion Agent 复用 normalize、chunk、embedding、sparse encode、publish、source monitor、审批回路这些底层能力。
复用检索原语
从 RAG Agent 复用 query embedding、hybrid retrieval、rerank、top-k merge、作用域过滤，但不复用完整 citation-heavy answer 流程。
保留独立生命周期
Memory 有自己的 daily memory、MEMORY.md、temporal decay、compaction writeback、dirty sync 和 graceful degradation，不应和知识库文档生命周期混为一体。
Memory Service 内部能力
Truth Source
`memory/{user_id}/daily/*.md` + `memory/{user_id}/MEMORY.md`，文件即真相源。
Write Path
用户显式记忆、会话摘要、compaction flush、网页源同步都先进入 Memory Writer，再触发索引更新。
Recall Path
优先做 scoped recall，再做 temporal decay、hybrid merge 和 prompt injection，不直接强耦合 RAG 回答器。
Sync Path
watcher + dirty flag + interval sync + graceful fallback，保证 memory 始终是活数据索引。
建议的复用边界
直接复用的能力
Chunker、Embedding Pipeline、BM25 Encoder、Hybrid Search、Reranker、Version Replace、Web Diff、Inbox 审批。
不要直接复用的整链路
普通 memory 写入不应默认走完整知识库重型入库链；memory recall 也不应默认走完整 RAG answer composition。
Compaction 的落点
Compaction 产生 durable memory 候选后，进入 Memory Writer，写回 Markdown truth source，再触发增量索引。
平台中的位置
Memory Service 应该位于支撑系统层，与 Knowledge Infrastructure 共用底座，并服务于 SensePedia Core 与各业务 Agent。
Self-Improving Agent 设计
LangGraph 负责编排，不会天然“自我进化”。平台需要额外构建一个反馈学习闭环，把用户显式反馈、隐式行为信号与执行轨迹沉淀为可验证的策略更新， 优先优化 Prompt、Tool Routing、Retrieval 参数与用户偏好记忆，而不是直接在线改模型权重。
1. Feedback Capture
从前端、API 和业务日志采集显式反馈与隐式反馈，形成统一反馈事件流。
  • 显式反馈：点赞、点踩、纠错、人工改写、满意度评分
  • 隐式反馈：是否继续追问、是否复制答案、是否采纳结果、任务是否完成
  • 上下文：用户 ID、线程 ID、Prompt 版本、工具链路、引用与耗时
2. Evaluation & Attribution
用 evaluator/critic 节点和离线分析，对问题根因进行归因，而不是把所有负反馈都归因给模型。
  • 检索不足：召回文档不对、topk 不够、过滤条件错误
  • 工具决策失误：选错 Tool、参数映射错误、执行顺序不合理
  • 答案质量不足：结构混乱、未结合证据、缺少引用或引用错误
3. Policy Update
将归因结果转为低风险、可回滚的策略更新，写入平台配置中心和长期记忆。
  • 更新 Prompt 模板、Few-shot 示例和输出 schema
  • 调整 Tool 优先级、Planner 路由规则、并行执行策略
  • 调节 Retrieval 参数、重排阈值、用户偏好配置
4. Safe Rollout
新策略必须先经过离线回放与灰度验证，再逐步放量，避免自我进化变成自我退化。
  • 历史样本回放、自动评测、人工抽检
  • 按租户/用户分桶灰度发布，比较成功率与 fallback rate
  • 策略版本化与一键回滚

平台内可落地的更新对象
Prompt Layer
针对高频差评场景维护 Prompt 版本、Few-shot 样例与答案模板，不直接改底模。
Planner Policy
根据成功率更新初始 action、follow-up tool 顺序、重试预算与路由规则。
Retrieval Policy
结合反馈调整 topk、过滤条件、rerank 阈值、知识域优先级和引用覆盖率。
Memory Service
把“用户喜欢的回答风格、常用知识域、容忍延迟、术语偏好”写入长期记忆与可召回记忆索引。
Self-Improving 配套平台能力
Feedback Store
持久化 feedback event、trace、prompt version、tool trace 与结果标签。
Evaluation Service
对低分回答做自动判分、根因分析和回放评测。
Policy Registry
维护可生效的 Prompt、Router、Retrieval、Memory 策略版本。
Experiment Manager
支持 A/B、canary、tenant-level rollout 和效果对比。
安全护栏
只允许更新“可解释、可回滚”的配置层，不允许 agent 直接在线修改核心代码和模型权重。
任何策略更新都必须绑定评估指标，如 grounded answer rate、tool error rate、citation coverage、user success rate。
将高风险领域的更新限制为“建议模式”，需要人工审批后才能进入生产。
用户隔离必须保留，单个租户或用户的反馈不能污染其他租户的默认策略。
LLM Planner 详细设计

本节描述的是 SensePedia Core 中的 LLM Planner 如何工作。它发生在选中某个业务 Agent 之后， 不负责业务 Agent 的选择；业务 Agent 选择由上游的 Agent Router 负责，LLM Planner 只负责当前 Agent 的内部执行规划。

Planner 核心功能
1. Task Analyzer (任务分析)
• 分析用户意图
• 识别任务类型
• 提取关键实体
2. Plan Generator (计划生成)
• 分解复杂任务为子任务
• 确定执行顺序
• 识别并行执行机会
3. ReAct Reasoning (推理行动)
• 思考 (Thought)
• 行动 (Action)
• 观察 (Observation)
4. Tool Selector (工具选择)
• 根据任务选择合适工具
• 参数映射
• 依赖分析

LLM Planner 与其他模块的交互关系
交互模块 交互方向 交互内容
Agent Router ─ 输入边界 Agent Router 先选定当前请求所属的业务 Agent，LLM Planner 不参与“选哪个 Agent”，只接收已选中的执行上下文。
Selected Business Agent ─ 所属上下文 LLM Planner 运行在当前被选中的业务 Agent 内部，继承该 Agent 的领域目标、工具范围与执行约束。
Tool Registry ─ 双向 查询可用工具列表、获取工具 schema、验证工具存在性
LLM Gateway ─ 输出 发送规划请求、接收 LLM 生成的计划 JSON
Graph Executor ─ 输出 传递执行计划、接收执行结果
Skills ─ 间接 通过 Tool Registry 间接调用具体 Skill
Knowledge Infrastructure ─ 间接 在知识库相关 Agent 中，LLM Planner 会围绕检索、入库、网页更新等任务规划对共享知识基础设施的调用顺序。
Memory Service ─ 双向 读取长期记忆与偏好上下文，并在需要时把新的 durable memory、摘要或 compaction 写回候选交给 Memory Service。
Observability ─ 输出 记录规划过程、耗时、决策依据
LLM Planner Prompt 示例 
你是 SensePedia Core 中的 LLM Planner。

你不负责选择业务 Agent。
当前请求已经经过 Agent Router，并且已经确定了 Selected Business Agent。
你的职责是在当前业务 Agent 的上下文内完成任务拆解、工具选择与执行规划。
