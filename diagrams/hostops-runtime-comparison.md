# Python LangGraph 场景下的 HostOps Runtime 方案对比

前提：

- 业务编排层：`Python LangGraph`
- 宿主机执行层：二选一
  1. 自研 `TS Runtime`
  2. 直接接入 `Pi`

| 维度 | 自研 TS Runtime | 直接使用 Pi |
|---|---|---|
| 核心定位 | 完全为平台定制的执行内核 | 现成的极简宿主机执行内核 |
| 上线速度 | 慢，前期研发成本高 | 快，适合先做 MVP |
| 架构控制力 | 最高，接口、状态、权限都可自定义 | 中等，需要适配 Pi 的模型 |
| 与现有系统契合度 | 最高，能贴合审批流、审计、租户隔离 | 中等，需要做桥接层 |
| 学习成本 | 中等，主要是自己设计边界 | 低到中等，先理解 Pi 的机制 |
| 长期可维护性 | 高，前提是团队持续维护 | 取决于 Pi 的演进和 fork 深度 |
| 二次开发灵活性 | 最高，想怎么改都行 | 有限，受 Pi 现有抽象约束 |
| 权限模型设计 | 可完全按平台的 RBAC / 审批模型设计 | 需要把现有权限体系映射到 Pi |
| 审批 / 中断 / 恢复 | 可按产品需求原生设计 | 需要看 Pi 的原生能力，不足部分要补 |
| 审计能力 | 可从第一天按企业级要求设计 | 通常需要外围补审计与留痕 |
| 工具接入方式 | 自定义 Tool Registry，最灵活 | 需要适配 Pi 的 tool contract |
| 与 Python LangGraph 集成 | 很顺，可直接定义 RPC / MCP / HTTP 协议 | 也能接，但通常要多一层适配 |
| 风险 | 研发周期长，容易过度设计 | 被上游设计牵制，深度定制时会别扭 |
| 适合阶段 | 平台化、长期建设、企业级落地 | 快速验证、早期试点、先跑通闭环 |
| 典型团队要求 | 需要稳定的后端 / 平台工程能力 | 小团队也能较快落地 |
| 总体结论 | **长期最优** | **短期最优** |

## 建议路线

- 想先验证产品价值：优先 `Pi`
- 已明确要做平台化、私有化、强审计、强权限：优先自研 `TS Runtime`
- 更现实的折中路线：**Pi 先行，自研预留接口**

## 推荐分层

```text
Python LangGraph
        │
        ▼
   HostOps API
        │
        ├── Pi
        └── Self-built TS Runtime
```

## 各层职责

- `Python LangGraph`：负责业务 Agent 编排、推理、任务拆解、工具选择
- `HostOps API`：负责能力抽象、权限校验、审批挂起、审计打点、错误模型统一
- `Pi / Self-built TS Runtime`：负责真正执行 `exec / ssh / files / process / browser / docker`

## Temporal 的位置

- `Temporal` 不是 Agent Brain，而是可选的 durable workflow backend
- 适合承载长任务、审批后恢复、重试、跨机器步骤编排
- MVP 阶段通常可以先不引入，后续在高可靠执行阶段再接入
