一、为什么这么火

OpenClaw 和之前的软件有什么不同

1
多渠道交互：可挂接 Telegram、WhatsApp、飞书等，天然贴近日常工作入口。
2
从“会聊”变成“会做”：具备 computer use 能力，可通过 Pi 这类执行框架操作宿主机和现有软件，不再局限于聊天框。
3
skills 机制可扩展：可以围绕具体工作流持续封装能力，而不是每次都从提示词重做。
4
memory 让它具备连续工作能力：不再像传统 LLM 一样“上下文一断就忘”。

技术原理，可用一句话概括

用户并不是在和一个聊天机器人对话，而是在和一套“能理解、能记住、能调用工具、能执行动作”的 Agent 平台交互。

二、案例

量化交易案例：Agent 不是替代系统，而是成为交互中心

痛
A 的工作分为人工下单和量化系统交易两部分，长期需要查多源数据、盯盘、处理风格切换和异常情况。
做
通过 OpenClaw + 多个 skill，把“抓因子、分析走势、下单、监控开平仓、推送告警”串成一个 Telegram Bot 工作流。
变
有了 Agent 之后，A 大幅降低直接登录交易所和量化系统的频次，只有触发紧急告警时才回原系统复盘。
结
关键变化不是功能替代，而是交互层重构。原有核心系统仍然存在，但日常入口变成了 Agent。

信号提醒：Agent 自动汇总策略判断、关键因子和风险提示。

交易反馈：平仓价格、盈亏和 ROI 直接回推到对话入口。

异常预警：当系统触发风险事件时，Agent 可以第一时间通知人工介入。

三、启示

从个人软件入口，走向企业工作流入口

量化交易背后的四层架构

④ OpenClaw 个人 AI Agent 平台新的人机交互层，接收指令、编排技能、汇总结果、发出通知。
③ 免费和付费数据源因子数据、新闻、价格、策略输入，仍然是基础信息来源。
② 量化交易系统策略、回测、风控、执行逻辑仍在原系统内部。
① 交易所软件底层交易通道和账户体系继续存在。

这个案例想说明的不是“Agent 会把所有旧系统干掉”，而是Agent 会逐步成为上层统一交互入口。

C
个人侧：用户不愿意在多个 App 和系统之间反复跳转。
B
企业侧：员工同样不愿意在 CRM、ERP、MES、BI 等多个系统中来回切换。
判
谁掌握跨系统工作流的 Agent 编排能力，谁更有可能占据下一代人机交互入口。

五、行业切法

从行业软件视角看，Agent 最有价值的地方是“跨系统工作流编排”

制造业

CRM (接单) → CAD/CAE (设计) → PLM (研发管理) → ERP (资源计划) → SRM (采购) → MES (生产执行) → QMS (质量) → WMS/TMS (仓储物流)

金融保险

CRM (获客/销售) → LOS/核保 (进件与审批) → 核心系统 (资金/保单处理) → 风控/合规 (风险管理) → ERP/EPM (财务与绩效) → BI/数据中台 (分析决策)

软件互联网

CRM/反馈 (需求收集) → PM 工具 (项目管理) → UI/UX (设计) → IDE/Git (开发) → CI/CD (测试与集成) → APM/云平台 (部署运维) → BI/分析 (运营数据)

房地产

投资测算 (拿地) → CAD/BIM (设计) → 招采系统 (采购) → 工程项目管理 (施工) → 售楼/CRM (销售) → 物业管理系统 (运营)

国防军工

MBSE (需求与指标) → CAD/CAE (涉密设计) → PDM/PLM (涉密研发管理) → ERP (军工计划) → MES (精密制造) → MRO (维修保障)

建筑工程

CRM/投标管理 (接单) → BIM/CAD (设计) → 造价/算量 (预算) → PMIS (项目管理) → 智慧工地 (现场执行) → 档案系统 (竣工验收)

零售电商

PLM/选品 (商品企划) → SCM/ERP (供应链) → PIM (商品信息) → OMS (订单管理) → POS/电商平台 (销售) → WMS/TMS (履约) → SCRM (会员营销)

医疗医院

预约挂号系统 (入口) → HIS (医院核心系统) → EMR (电子病历) → LIS/PACS (医技检查) → 药房系统 (发药) → 医保结算 (支付) → 随访/CRM (院后)

几乎所有重点行业都有一个共同点：系统很多、角色很多、流程很长、数据跨系统流转。

这正是 Agent Platform 的机会所在。它最先替代的不是 ERP、MES、HIS 本身，而是员工在多个系统之间不断切换、查询、录入、催办和跟踪的那层工作方式。

六、落地路径

面向 ToB / ToG 的推进思路：先做编排层，再争取入口层

①
打造一套 AI Agent Platform

先把基础能力搭好，包括模型接入、memory、skill、权限、审计、通知和多端入口。

②
深入一个战略性行业

建议先从制造业切入，因为流程长、软件多、价值密度高，也最能证明平台能力。

③
梳理工作流和系统数据流

不是只看功能菜单，而是要看订单、BOM、计划、采购、生产、质量、物流数据如何流转。

④
联合核心软件厂商做 SDK / skill

通过官方接口，把原来页面上“点点点”的动作沉淀成可调用的业务能力。

⑤
先替代低效协同与 RPA

先吃掉查询、跟进、催办、填单、通知、复核这层重复劳动，再向闭环执行延展。

⑥
从助手演进为入口

当一个行业里的关键系统越接越多，平台才有机会成为真正的工作流交互中心。

一张图总结定位变化

阶段 1：助手层帮人查信息、调工具、发通知。
阶段 2：编排层串联多个系统，让流程跨系统跑起来。
阶段 3：入口层员工主要通过 Agent 平台完成工作，旧系统退到后台能力层。

七、挑战

真正难的，不是把模型接进来，而是把 Agent 嵌入真实业务流程

1. 系统异构，集成成本高

以制造业为例，一个完整流程要串起 CRM、PLM、ERP、SRM、MES、QMS、WMS/TMS，多数系统高度定制，接口标准不统一，很多甚至没有成熟 SDK。

2. 安全合规和责任边界极高

金融、医疗、军工、政府场景里，很多动作不能黑盒自动执行，必须可审计、可追溯、可回滚，还要保留人工审批。

3. 企业数据质量并不天然适合 Agent

客户、订单、BOM、病历、保单往往分散在多个系统，定义口径不统一。底层数据不干净，Agent 越强，放大的风险越大。

4. 商业生态未必愿意开放

要成为入口，就需要核心软件厂商开放接口与能力。但对方也会担心，入口和客户关系被 Agent 平台重构，合作博弈会很复杂。

5. 产品化与项目化之间有张力

如果大量依赖定制 skill 落地，短期可以做项目，长期却未必能形成标准化平台和可复制收入。

因此，路径判断

这个方向机会很大，但不能一上来就讲“全面替代”。更现实的打法是：先从高价值、可闭环、强刚需流程切入，逐步从助手变成编排层，再争取成为入口层。

八、收口

结论

我的判断不是企业软件会被立刻替代，而是企业软件的使用方式会被重构。谁先掌握工作流级 Agent 能力，谁就更有机会占据下一代 B 端 / G 端的人机交互入口。

一句话结论

OpenClaw 火的本质，是它把大模型从“会回答”推进到“会执行”。

一句话机会

ToB 机会在于成为跨系统工作流的交互层和编排层。

一句话挑战

最大的难点不是模型，而是集成、合规、数据和可复制性。
