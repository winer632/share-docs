基于 OpenClaw + Gemini 3.1 Pro 的视频事件检索与目标框选方案

目标：从约 3GB 视频中检索出符合 description.jpg 描述的目标/行为，返回对应时间点帧图，并输出类似 frame1.png 的高质量画框结果。

推荐架构：YOLO 预过滤 + Gemini 语义召回 + 精定位 适配场景：离线审查 / 准实时回放分析

1. 最终推荐（结论先行）

最合理的方案不是“纯 Gemini”也不是“纯 YOLO”，而是分层混合：

  1. 本地轻量预过滤： YOLO(person) + 运动检测 + 简单人群统计，快速剔除空帧和低价值片段。
  2. 语义时间召回： 对候选片段调用 Gemini 3.1 Pro，找“事件发生时间窗”（而非直接像素级最终框）。
  3. 空间精定位： 对时间窗内关键帧使用 Grounding DINO / YOLO(定制) + 跟踪器进行精框与连续性校验。
  4. 规则引擎判定： 将“其他人坐着、仅一人站立并举手”等复杂关系写入可解释规则层。
  5. 渲染输出： 回到原视频时间点截帧，画框、标注置信度、时间戳和规则命中原因。
这条链路在准确率、成本、吞吐和可维护性上最均衡，且与 OpenClaw 的 Agent 编排模式天然匹配。

2. 为什么这样设计

环节 只用 Gemini 的问题 只用检测模型的问题 混合方案收益
粗筛 成本高、耗时高 能筛“有人”，但筛不出复杂语义 本地模型快速降样本，减少 API 调用量
事件理解 可做，但如果全量扫会慢 关系语义弱（例如“只有一人站立且举手”） Gemini 专注“时间定位 + 语义解释”
精准框选 坐标稳定性依赖模型与模式 擅长框选，但不懂复杂语义上下文 检测器负责几何精度，LLM 负责语义判断
可解释性 黑盒风险高 规则缺失时难解释 规则引擎输出命中依据，便于审计

5. 关键算法与参数建议

5.1 采样策略

  • 全视频粗采样：0.5~1 FPS（用于预过滤统计）。
  • 候选窗精采样：4~8 FPS（用于动作细节和跟踪）。
  • 高速动作（如“快速奔跑/拔取动作”）：局部可升到 10~15 FPS。

5.2 规则引擎示例（“其他人坐着，独自站立并举手”）

  1. 检测到人数 N >= 3。
  2. 姿态估计得到“站立人数 == 1”。
  3. 该站立者手部关键点满足“双手高于肩线”持续 T >= 1.0s。
  4. 其余人员坐姿比例满足 >= 70% 且持续 T >= 1.0s。
  5. 跟踪 ID 连续存在，排除瞬时误检。

5.3 抗幻觉策略

  • Gemini Prompt 固定要求：无证据时返回 NOT_FOUND。
  • 同一事件需满足“语义召回 + 空间定位 + 规则引擎”三重一致。
  • 输出附带证据字段：timestamps、bbox_seq、pose_score、reason。
不建议把“LLM 文本回答”直接当最终检测结果。必须经过检测器和规则层复核。

7. 工程实现要点（非常关键）

7.1 不要默认先压 720p

  • 优先：分片 + 保清晰抽帧。
  • 仅在带宽/上传限制需要时再进行有控制压缩。
  • 可采用 ROI 裁剪（看台区域、通道区域）替代全局降分辨率。

7.2 性能与成本控制

  • 批处理：按分片并行跑预过滤。
  • 缓存：同片段语义搜索结果缓存，避免重复调用。
  • 阈值门控：低运动、低人数变化窗口直接跳过 Gemini。
  • Top-K 策略：仅将高可疑窗口送入精定位。

7.3 质量评估体系

8. 分阶段落地计划

阶段 目标 输出 验收标准
P0（1-2周） 跑通端到端 可对单条行为检索+出框图 手工验证样本可复现结果
P1（2-4周） 支持多行为并行 规则引擎 + 批处理流水线 事件召回率达到业务可用阈值
P2（4-8周） 成本优化与稳定性 缓存/重试/监控/告警 单位视频成本下降，误报下降

9. 示例工作流伪代码（可直接映射到 OpenClaw）

def pipeline(video_path, action_defs):
    segments = video_preprocess(video_path, split_minutes=10, keep_resolution=True)

    candidates = []
    for seg in segments:
      c = prefilter_yolo_motion(seg, fps=1)
      candidates.extend(c)

    semantic_hits = []
    for action in action_defs:
      semantic_hits.extend(
        gemini_temporal_search(candidates, action.prompt, enforce_not_found=True)
      )

    final_events = []
    for hit in semantic_hits:
      keyframes = frame_extract(video_path, hit.start_sec, hit.end_sec, fps=6)
      dets = spatial_localize(keyframes, hit.localization_prompt)
      tracks = track_objects(dets)
      verdict = rule_engine(action=hit.action_id, tracks=tracks, min_duration=1.0)
      if verdict.hit:
        artifact = render_result(video_path, verdict.best_ts, verdict.best_bbox, hit.action_name)
        final_events.append({**verdict, "artifact": artifact})

    return final_events

10. 风险与对策清单

  • 小目标看不清： 不先全局降采样，优先 ROI + 高分辨率关键帧。
  • 快动作漏检： 候选窗提高 FPS，增加连续帧判定。
  • 误报偏高： 引入规则持续时间阈值、跨模型一致性校验。
  • 成本过高： YOLO/motion 预过滤 + Top-K + 缓存。
  • 模型波动： 多模型兜底与 A/B 评估，不绑定单一供应商能力。

11. 你当前项目的推荐默认配置

  • 场景类型：离线审查（非硬实时）。
  • 视频处理：10 分钟分片，原分辨率保留。
  • 预过滤：YOLO(person) + motion，粗采样 1 FPS。
  • 语义召回：Gemini 仅分析候选片段。
  • 精定位：Grounding DINO + ByteTrack，关键窗 6 FPS。
  • 输出：best_frame.jpg、annotated_clip.mp4、event_report.json。
这套默认配置可以先快速上线，后续再按数据集迭代阈值和模型组合。
